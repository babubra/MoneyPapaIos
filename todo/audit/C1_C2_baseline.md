## Summary

После реализации M6 (usage-логирование) измерены реальные **baseline-цифры** в проде: `prompt ≈ 3300, completion ≈ 180, cached = 0` на каждый запрос. Это подтверждает главную дыру C1.1: SYSTEM_PROMPT (~3000 stable + ~300 dynamic токенов) платится полностью на каждом запросе, prompt-cache **не работает** на текущей модели.

Анализ цен aitunnel и доки показал, что **cache на текущей `gemini-2.5-flash-lite` не активируется** (aitunnel поддерживает auto-cache только для 2.5 Pro / 2.5 Flash, и эмпирические данные `cached=0` это подтверждают). Перевод на `gemini-2.5-flash` + cache **не окупается** даже при 90% hit rate — Flash в 2-3× дороже Lite по output-токенам.

**Самая большая экономия** — не в смене модели, а в **M13 (сокращение SYSTEM_PROMPT)**: -38% input-токенов одной правкой промпта, на любой модели. Гибридная стратегия (text → китайская/Qwen, audio → Gemini Lite) даёт дополнительные ~22-40% экономии, но **только после M14 (eval-набор)**, иначе мы рискуем потерять качество русской финансовой лексики на 12 локалях.

Замеры с реальных запросов от 2026-05-07, user_id=19:

| Тип запроса | prompt | completion | total | cached | input размер |
|---|---|---|---|---|---|
| **Text** (77 символов, 2 категории) | 3293 | 190 | 3483 | 0 | text 77 chars |
| **Audio** (≈30 сек, 35.7 KB m4a) | 3314 | 170 | 3484 | 0 | binary 35.7 KB |

### Что говорят числа

1. **Аудио НЕ попадает в `prompt_tokens`.** Разница prompt audio vs text — всего 21 токен (3314 vs 3293), хотя там 30 секунд аудио. Gemini считает аудио отдельно (или скрыто). Это значит мы **не видим истинную стоимость аудио** через `usage` — нужна ручная проверка цены за аудио в aitunnel-биллинге.
2. **3293 токенов на 77-символьный ввод + 2 категории** — пользовательский input занимает ~50 токенов, остальные **~3240 = SYSTEM_PROMPT + base scaffolding**. Соотношение сигнал/шум = 1.5%.
3. **`cached = 0` на оба запроса** в пределах 2-минутного окна (Gemini implicit-cache TTL = 3-5 мин). Подтверждает, что **prompt-cache на flash-lite не работает** через aitunnel.

### Артефакт для дальнейшей разведки

[`backend/check_cache.py`](../../backend/check_cache.py) — standalone-скрипт; рекомендуется запустить вручную после смены модели и распечатать `usage` целиком (`print(usage)`), чтобы увидеть, есть ли в провайдерском ответе скрытые поля (`audio_tokens`, `cached_tokens`, etc.).

## Реалистичный cache hit rate для нашего профиля

Implicit Gemini-кэш живёт **3-5 минут**. Профиль финансового приложения:
- Обычный юзер: 1-3 ввода подряд после магазина / получки → потом часы тишины.
- Активный юзер: 5-10 транзакций за вечер → возможен hit rate ~50-60% во время сессии.
- В среднем по DAU реалистично ожидать **20-40% hit rate**, не 70-90%.

**Поэтому в реальных деньгах cache даёт меньше, чем «оптимистичный» сценарий из таблиц.**

## Гибридная стратегия (text + audio split)

Идея: разделить роутинг — **text → дешёвая китайская модель**, **audio → Gemini** (китайцы audio не принимают через aitunnel).

### Преимущества

1. **Text-парсинг ≈ 70% наших AI-запросов** (по интуиции — нужно подтвердить). Экономия 22-40% на нём = реальные деньги.
2. **Китайские модели новее.** Qwen 3 / GLM 4.7 — конец 2025, актуальные. `gemini-2.5-flash-lite` — июнь 2025, начинает «тупить» на сложной JSON-схеме (твой опыт).
3. **Audio остаётся на Gemini Lite** — самая дешёвая среди audio-capable моделей. Даже GPT Audio Mini в 6× дороже.

### Реализация (если M14 подтвердит качество)

1. Разделить env-переменные:
   ```
   AI_MODEL_TEXT=qwen3-235b-a22b-2507  # или GLM 4.7 Flash
   AI_MODEL_AUDIO=gemini-2.5-flash-lite  # или Gemini 2.0 Flash Lite 001
   ```
2. В `_call_ai_text` — `model=settings.AI_MODEL_TEXT`, в `_call_ai_audio` — `model=settings.AI_MODEL_AUDIO`.
3. SYSTEM_PROMPT можно оставить **общий** — он provider-agnostic. При желании потом можно сделать tweaks под конкретного провайдера (например, Qwen лучше реагирует на инструкции на английском).
4. Через ту же aitunnel-base_url — менять не нужно.

### Расчёт экономии (грубо, при 70% text / 30% audio распределении и 1000 запросов/день)

| Сценарий | Cost/day (₽) | vs текущая |
|---|---|---|
| Текущая: всё на Lite (1000 × 0.077) | 77 | 1.00× |
| Hybrid: text Qwen 235b (700×0.060) + audio Lite (300×0.077) | 65 | 0.84× |

**Потенциальная экономия 16-36% / день**, не считая выигрыша от M13.

### Риски китайских моделей конкретно для нас

1. **Локализация:** SYSTEM_PROMPT поддерживает 12 локалей (ru/en/de/fr/es/it/pt/tr/zh/ja/ko/ar). Qwen/GLM хорошо натренированы на zh+en, **на ru** — ОК-уровень, на финансовой лексике («получка», «налик», падежи числительных, «алименты», «зарплатная карта»...) могут давать ошибки 5-10%. На редких локалях (de/fr/ar) — больше.
2. **Auto-learn маппинги:** SYSTEM_PROMPT раздел 5 настойчиво говорит модели сначала смотреть user-маппинги. Слабые модели игнорируют этот раздел. Если Qwen 30b A3b не уважает маппинги — мы теряем главную ценность auto-learn.
3. **Off-topic detection:** на «какая погода?» текущий промпт возвращает `status="off_topic"`. Китайская модель может попытаться парсить как-то всё равно — нужно проверять.
4. **Strict json_schema** (если будем переходить на M9): не везде гарантированно работает. JSON mode (`response_format=json_object`) поддерживают почти все, strict — нет.

**Минимальный критерий M14 для перехода:** ≥ 85% accuracy на ru+en для type/amount/category. Если ниже — отказ.

### Как проверить эмпирически

1. **На сайте aitunnel — теги в карточке модели.** `Tools` = function calling/structured output, надёжный сигнал.
2. **curl-тест** — послать запрос с `response_format={"type":"json_schema","json_schema":{...}}` через aitunnel:
   - 200 + JSON по схеме → strict работает
   - 200 + JSON с лишними/пропущенными полями → только object mode
   - 400 / unknown_field → не поддерживает вообще
3. **Документация провайдера**: Qwen [docs](https://qwen.readthedocs.io/), GLM [bigmodel](https://docs.bigmodel.cn), DeepSeek [api-docs](https://api-docs.deepseek.com).

## Что нужно для M14 (eval-набор)

**Минимальный набор для принятия решения по смене модели:**

- **30 примеров** ru транзакций (income/expense/долги/контрагенты/категории; короткие/длинные; с категорией/без; в формате auto-learn-маппинг сценария).
- **15 примеров** en — для проверки multilingual.
- **5 примеров** off-topic («какая погода?», «привет», «как дела») — для проверки `status="off_topic"`.
- **5 примеров** edge cases: пустые категории, 100+ категорий (DoS-симуляция для C2.7), очень длинный текст (граница `AI_MAX_TEXT_LENGTH=500`), смешанные языки в одном запросе.
- **Ground truth** для каждого: ожидаемый JSON (type, amount, currency, category_match, status).

**Процесс прогона:**
1. Прогнать все 55 примеров через каждую модель-кандидат через aitunnel.
2. Сравнить ответы с ground truth по полям.
3. Метрики:
   - **Accuracy** (% правильных type/amount/category)
   - **JSON validity rate** (% запросов без retry)
   - **Latency** p50 / p95 / p99
   - **Cost per correct result** (а не per request — это интегрирующая метрика)

**Кандидаты для прогона (приоритет по последовательности):**
1. `gemini-2.5-flash-lite` (текущий baseline)
2. **Qwen3 235b A22b 2507** — главный кандидат на text (большая, дешевле Lite на 22%)
3. **GLM 4.7 Flash** — если Qwen 235b не показал ≥85% (дешевле, но риск качества выше)
4. **Gemini 2.0 Flash Lite 001** — для audio downgrade (-25% от текущей цены)
5. **Gemini 3.1 Flash Lite** — если 2.5-Lite реально проседает по качеству, но цена в 2.7× — оправдает только при значительном росте accuracy

## Что нужно уточнить у aitunnel перед окончательным решением

1. **Точная цена `cached input` для Gemini-моделей** — не показана в UI. От этого зависит, насколько Flash+cache подъезжает к Lite. (Если у aitunnel cached=0.10× как у OpenAI, не 0.25× как у Google — Flash при 70% hit может стать на ~12% дешевле Lite.)
2. **Цена audio input** — у Google аудио в 13× дороже текста (≈$1/1M против $0.075/1M для Flash). aitunnel может тарифицировать отдельно или прятать в общем тарифе. Для нашего audio-сценария это критично.
3. **GPT-5 Mini акционная цена 4.8 ₽** — постоянна или временна? Базовая после акции?
4. **Поддерживает ли aitunnel `response_format=json_schema`** на Gemini-моделях? (Технически Gemini нативно требует `responseSchema`, OpenAI-compat прокси может транслировать, может нет.)
5. **Есть ли у aitunnel `gpt-5.4-nano`** — это потенциально единственная OpenAI-модель с экономикой ≈ Lite (по OpenAI-base $0.20/M ≈ 19 ₽/M).

## Главные рекомендации (в порядке убывания приоритета)

1. **Не менять модель сейчас. Без M14 (eval-набор) любая смена — слепая.**
2. **Быстрые caps — лёгкий бэйкенд** (часовая работа суммарно):
   - **C2.10** (5 мин): `max_tokens=1024` в `_call_ai_audio` — защита от output-overflow.
   - **C1.5** (15 мин): `LIMIT settings.AI_MAPPINGS_PROMPT_LIMIT=30` в `_load_user_mappings` + конфиг + cleanup endpoint.
   - **C2.7** (10 мин): `max_length=200` на `categories`/`counterparts` в `ParseTextRequest` + хвост-обрезка с WARNING.
3. **M14 (eval-набор)** — 55 примеров с ground truth в `backend/tests/test_ai_parse.py`, фикстуры через `respx`. Без M14 нельзя осмысленно мерять следующее.
4. **M13 (сокращение SYSTEM_PROMPT)** — после M14. Это самая большая прямая экономия (~38% input-токенов на любой модели).
5. **Гибридная стратегия `text → Qwen 235b, audio → Gemini Lite`** — после M14, через A/B с baseline. Если Qwen 235b ≥ 85% accuracy на ru/en — переключаемся; иначе пробуем GLM 4.7. Дополнительная экономия 22-40% поверх M13.
6. **Параллельно — закрыть остальные C1+C2-fixups:** retry/идемпотентность на клиенте (C1.3), Premium daily-cap (M15), validation locale (C2.21), убрать мёртвый rate-limit конфиг (C2.17).

## Что НЕ покрыли в этой сессии

- **Не запустили `check_cache.py`** против текущей модели для подробного `usage`-объекта (сырой dump). Без него мы не видим, есть ли скрытые поля типа `audio_tokens`. Полезно сделать как один curl-эксперимент.
- **Не проверили эмпирически response_format=json_schema** на разных моделях через aitunnel. Это тест на 5 минут (один curl на каждую кандидат-модель).
- **Не замерили реальный профиль text/audio** в проде — пока работаем на интуиции «70% text / 30% audio». Через несколько недель использования из M6-логов можно посчитать точно.
- **Не оценили качество Gemini 3.1 Flash Lite** vs 2.5 Lite. Цена в 2.7× выше, но если она реально умнее — может закрыть проблему «2.5 Lite тупит» без перехода на китайцев. Эксперимент с eval-набором ответит.
- **Не учли StoreKit-будущее.** При появлении Premium-подписок мы получим Premium-юзеров без daily-cap (M15) — их usage может скакать в 10× от free. Эта дисперсия повлияет на cache hit rate.
- **Не оценили эффект streaming** (`stream=True`). Для UX мог бы сократить «время до первого токена» в голосовом флоу, но требует переписывания клиента + partial-JSON parser. Не критично.
